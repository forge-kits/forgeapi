# Аудит: Test Suite

## Краткое резюме

Тестовый набор ForgeAPI (108 тестов в 9 тестовых файлах) хорошо структурирован для модулей, которые он охватывает: используется anyio, автоматически применяемые фикстуры для сброса синглтонов, а также юнит- и интеграционные уровни. Однако имеются значительные пробелы и ряд проблем с качеством.

Критические пробелы: модуль авторизации (стратегии JWT/Cookie/Telegram, AuthBackend, зависимость CurrentUser) полностью не покрыт тестами, хотя является наиболее чувствительным с точки зрения безопасности кодом. Модуль разрешений (PermissionsMixin, CRUD для ролей и разрешений, патчинг сигнатур Guard) не имеет никакого покрытия. RedisBus (автономная шина публикации/подписки между проектами) не протестирован. Также не покрыты EventBus.load_from_dir/_import_file, start_redis_subscriber и логика дедупликации.

Проблемы качества: test_background_dispatch_fires использует нестабильный spin-loop через asyncio.sleep(0), который может давать ложные срабатывания. test_listener_exception_does_not_propagate не проверяет, что ошибка была залогирована. test_logs_request проверяет только два фрагмента строки в выводе лога. test_redis_dispatch_publishes_not_local не валидирует содержимое JSON-нагрузки. test_from_attributes_enabled пропускает проверку полей created_at/updated_at.

Пробелы в граничных случаях: Event.from_dict с отсутствующим ключом _event_type; синхронный обработчик, зарегистрированный на асинхронной шине; RateLimitMiddleware с rpm=0 или отрицательным значением; Paginator.configure(default_limit > max_limit); load_config с некорректным TOML; _pluralize с пустой строкой; CORSMiddleware helper; Guard middleware.

Риски фикстур: модульный FastAPI-приложение в test_pagination.py совместно используется всеми интеграционными тестами и создаётся до срабатывания autouse-фикстур; @listen на уровне модуля в тестовых файлах пережил бы EventBus.reset(); фикстура anyio_backend молча пропускает тестирование совместимости с trio.

### [HIGH] Fixture Issue - Module-Level Shared State — test_pagination.py использует FastAPI-приложение на уровне модуля, что вызывает загрязнение состояния между тестами при изменении class vars Paginator

**Файл:** tests/test_pagination.py:13

**Серьёзность:** high

**Описание:**
Экземпляр `app` создаётся во время импорта модуля с фиксированным замыканием над классом Paginator. Тесты в TestPaginatorConfigure изменяют Paginator.DEFAULT_LIMIT и Paginator.MAX_LIMIT. Хотя conftest.py сбрасывает их после каждого теста, эндпоинт модульного приложения читает class vars во время вызова, поэтому состояние корректно. Однако модульный `app` разделяется между всеми httpx-тестами, и если любой будущий тест изменит приложение (например, добавит middleware или переопределит), это затронет все интеграционные тесты. Более критично то, что приложение создаётся до того, как может сработать фикстура reset_paginator, поэтому если какой-либо тест вызовет configure() до интеграционных тестов, значения по умолчанию, видимые httpx-тестам, могут отличаться от ожидаемых.

```python
app = FastAPI()

@app.get("/items")
async def list_items(pagination: Pagination):
    ...
```

**Рекомендация:**
Перенести создание тестового FastAPI-приложения внутрь pytest-фикстуры (с областью видимости 'function') вместо уровня модуля. Это гарантирует, что приложение создаётся заново для каждого теста с известными значениями class vars по умолчанию.

### [HIGH] Async Test Issue - Unawaited Background Task — test_background_dispatch_fires использует spin-loop через asyncio.sleep(0) — нестабильно и может давать ложные срабатывания

**Файл:** tests/test_events.py:195

**Серьёзность:** high

**Описание:**
Тест передаёт управление через `for _ in range(5): await asyncio.sleep(0)`, чтобы позволить фоновому create_task выполниться. Это эвристика, которая может давать сбои в медленном CI или при изменении планировщика asyncio. Если задача занимает больше времени, чем 5 итераций передачи управления, утверждение молча проходит с пустым списком. Также нет утверждения, что `received` содержит ровно один элемент — только что он равен [7], что завершится ошибкой, если обработчик выполнился более одного раза, но не если он не выполнился вообще до окончания цикла.

```python
for _ in range(5):
    await asyncio.sleep(0)
assert received == [7]
```

**Рекомендация:**
Использовать asyncio.wait_for с коротким таймаутом вокруг условия, или использовать anyio TaskGroup/CancelScope. Альтернативно, получить задачу из _bg_tasks и явно её дождаться: `bus._bg_tasks` является множеством, поэтому после диспетчеризации можно итерировать и ожидать задачи.
