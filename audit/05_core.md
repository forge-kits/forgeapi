# Аудит: Core (exceptions, kit, config, pagination, controllers, schemas, settings, seeder)

## Краткое резюме

Аудит 8 основных модулей ForgeAPI выявил 18 находок, охватывающих проблемы безопасности, корректности, производительности и архитектуры. Наиболее критичные проблемы: пустой JWT-секрет, который молча допускает подделку токенов (`kit.py/_build_jwt`); захардкоженный пароль по умолчанию «admin123» в docstring Seeder, который будет бездумно скопирован в продакшн; и отсутствие обёртки транзакции в базовом классе Seeder, что допускает частичное состояние базы данных после неудачного посева. Высокосерьёзные проблемы включают: изменяемые дефолты уровня класса на Controller (общие `tags`/`guards` между подклассами), `Paginator.configure()`, мутирующий глобальное состояние класса без блокировки (потокобезопасность), маршрутные обработчики контроллера, привязанные к одному экземпляру (устаревшие замыкания при перезагрузке), и Seeder, не бросающий `NotImplementedError` как типизированный сигнал. Среди проблем среднего уровня: отсутствие валидации значений `PaginationConfig` в `config.py`, отсутствие верхней границы для `limit` даже после ограничения `max_limit` (переполнение целых чисел в арифметике смещения при экстремальных страницах), N+1 паттерн COUNT+SELECT в документации пагинатора, `BaseUpdateSchema` без принудительного частичного обновления, `BaseAppSettings`, логирующий чувствительные поля, и `load_config`, молча игнорирующий неизвестные ключи верхнего уровня в TOML. Находки низкого и информационного уровня охватывают: отсутствие HTTP-кодов статуса в подтипах `ForgeAPIError`, отсутствие `__all__` в exceptions, угловые случаи `_pluralize` и отсутствие валидатора формата `base_prefix`.

## Находки

### [CRITICAL] [Security] — JWT secret молча откатывается к пустой строке, допуская подпись токенов без секрета

**Файл:** `kit.py:173`
**Серьёзность:** Критическая

**Описание:**
`_build_jwt()` вызывает `os.getenv(cfg.jwt_secret_env, "")` и передаёт результат напрямую в `JWTStrategy` без какой-либо проверки на непустое значение. Если переменная окружения отсутствует, `secret_key=""` передаётся дальше. `JWTStrategy.__init__` имеет защиту от этого, но она срабатывает только при инициализации `JWTStrategy` с пустой строкой — опасность заключается в том, что ошибка возникает во время запуска запроса, а не перехватывается в `kit.py`. В некоторых неправильно настроенных окружениях само имя переменной (`jwt_secret_env`) может быть написано с опечаткой, что приведёт к молчаливому игнорированию проблемы. Что ещё критичнее: если пользователь библиотеки использует `Core(auth="jwt")` и имя env-переменной в TOML написано неверно, система стартует с пустым секретом и подписывает настоящие токены им.

```python
secret_key=kwargs.pop("secret_key", os.getenv(cfg.jwt_secret_env, ""))
```

**Рекомендация:**
После получения секрета валидировать его перед передачей дальше: `secret = os.getenv(cfg.jwt_secret_env, ""); if not secret: raise ForgeAPIConfigError(f"Environment variable '{cfg.jwt_secret_env}' is not set.", hint="Set it before starting the server.")`. Дополнительно валидировать, что `jwt_secret_env` является непустой строкой в `AuthTomlConfig` с помощью field validator в Pydantic.

### [CRITICAL] [Security] — Docstring захардкоживает пароль 'admin123', нормализуя небезопасные паттерны в Seeder

**Файл:** `database/seeder.py:17`
**Серьёзность:** Критическая

**Описание:**
Канонический пример использования в docstring Seeder использует литеральную строку «admin123» в качестве значения пароля администратора. Разработчики, незнакомые с фреймворком, будут бездумно копировать этот фрагмент напрямую в продакшн-сидеры. Это антипаттерн безопасности на уровне документации, который исторически приводил к тому, что продакшн-системы поставлялись с дефолтными учётными данными.

```python
"password_hash": hash_password("admin123"),
```

**Рекомендация:**
Заменить захардкоженный пример пароля ссылкой на переменную окружения: `"password_hash": hash_password(os.environ["ADMIN_PASSWORD"])`. Добавить комментарий с предупреждением, что захардкоженные учётные данные никогда не должны попадать в продакшн.

### [HIGH] [Data Integrity] — Seeder.run() не имеет обёртки транзакции, допуская частичное состояние посева

**Файл:** `database/seeder.py:25`
**Серьёзность:** Высокая

**Описание:**
Базовый класс Seeder не предоставляет контекст транзакции. Каждый вызов `seed_cmd._execute()` вызывает `seeder_cls().run()` вне какой-либо транзакции базы данных. Если сидер создаёт несколько записей и завершается с ошибкой на полпути (например, нарушение ограничения уникальности на третьей из пяти записей), база данных остаётся в частичном состоянии без отката. Повторный запуск сидера столкнётся с частично существующими данными и может каждый раз завершаться с разными ошибками.

```python
async def run(self) -> None:
    raise NotImplementedError
```

**Рекомендация:**
Обернуть каждый запуск сидера в атомарную транзакцию на уровне фреймворка. Либо переопределить шаблон в `seed_cmd._execute()` с `async with in_transaction(): await seeder_cls().run()`, либо добавить финальный метод `execute()` в Seeder, который оборачивает `run()` в `tortoise.transactions.in_transaction()` и вызывать его из CLI-раннера. Задокументировать границу транзакции в docstring класса.

### [HIGH] [Correctness] — Изменяемые дефолты класса (tags, guards) разделяются между всеми подклассами Controller

**Файл:** `controllers/base.py:55-56`
**Серьёзность:** Высокая

**Описание:**
Controller объявляет `tags: list[str] = []` и `guards: list = []` как атрибуты уровня класса с изменяемыми дефолтными значениями в виде списков. Когда выполняется `__init_subclass__`, он читает `cls.__dict__.get("guards")`, но запасной вариант — `[]` — это тот же объект пустого списка, разделяемый базовым классом. Если подкласс добавляет элементы в `guards` во время выполнения без повторного объявления атрибута, это мутирует разделяемый список базового класса, затрагивая все другие подклассы. Аналогично, проверка `if "tags" not in cls.__dict__ or not cls.tags` читает `cls.tags`, который разрешается к атрибуту базового класса через MRO, если не переопределён.

```python
tags: list[str] = []
guards: list = []
```

**Рекомендация:**
Использовать `None` в качестве сигнального значения и создавать новые списки для каждого класса: `tags: list[str] | None = None` и `guards: list | None = None`. В `__init_subclass__` всегда присваивать новый список: `cls.tags = list(cls.__dict__.get("tags") or [])` и `cls.guards = list(cls.__dict__.get("guards") or [])`.

### [HIGH] [Correctness] — Маршрутные обработчики контроллера привязаны к одному экземпляру, нарушая изоляцию состояния

**Файл:** `controllers/base.py:93-99`
**Серьёзность:** Высокая

**Описание:**
При первом вызове `__init__` он итерируется по `dir(cls)`, получает привязанные методы через `getattr(self, name)` и регистрирует их в маршрутизаторе с использованием метода, привязанного к конкретному экземпляру. Флаг `_registered` на уровне класса затем предотвращает повторную регистрацию. Любое последующее создание экземпляра того же подкласса Controller (например, при разборке/настройке тестов или горячей перезагрузке) будет использовать методы исходного экземпляра, навсегда привязывая маршрутизатор к одному объекту. Это также означает, что любое состояние экземпляра, изменённое во время запроса, утекает в последующие запросы.

```python
cls.router.add_api_route(
    meta["path"],
    getattr(self, name),  # bound to this specific instance forever
    methods=meta["methods"],
    **meta["kwargs"],
)
```

**Рекомендация:**
Регистрировать несвязанные методы класса и позволить FastAPI обрабатывать создание экземпляров на каждый запрос, или сделать контроллеры полностью без сохранения состояния. Если контроллеры с состоянием задуманы намеренно, чётко задокументировать природу синглтона и добавить предупреждение о том, что состояние экземпляра разделяется между всеми запросами.

### [HIGH] [Thread Safety] — Paginator.configure() мутирует разделяемое состояние класса без блокировки

**Файл:** `pagination/paginator.py:62-76`
**Серьёзность:** Высокая

**Описание:**
`configure()` является classmethod, который напрямую записывает в `cls.DEFAULT_LIMIT` и `cls.MAX_LIMIT`. В многопоточном сервере (например, gunicorn с `--workers` и потоками, или набор тестов, выполняющих асинхронные тесты параллельно), вызов `configure()` в одном потоке может частично обновить класс, пока другой поток находится в середине выполнения `__init__`, читая старый `DEFAULT_LIMIT` и новый `MAX_LIMIT`, что приводит к несогласованному экземпляру Paginator.

```python
cls.DEFAULT_LIMIT = default_limit
cls.MAX_LIMIT = max_limit
```

**Рекомендация:**
Задокументировать, что `configure()` должен вызываться только при запуске до того, как приложение начнёт обрабатывать запросы (что является текущим паттерном использования через `Core.__init__`). Альтернативно, защитить запись с помощью `threading.Lock` на уровне класса. Также добавить валидацию того, что `default_limit <= max_limit`: `if default_limit > max_limit: raise ValueError("default_limit must not exceed max_limit")`.

### [HIGH] [Performance] — Пример в docstring демонстрирует антипаттерн N+1 COUNT + SELECT

**Файл:** `pagination/paginator.py:29-32`
**Серьёзность:** Высокая

**Описание:**
Официальный пример использования в docstring Paginator показывает два отдельных ожидаемых запроса: `total = await Product.all().count()` с последующим `items = await Product.all().order_by(...).offset(...).limit(...).values()`. Это канонический N+1 паттерн на уровне пагинации: каждый пагинированный эндпоинт выполняет два независимых SQL-запроса. При масштабировании это удваивает количество обращений к базе данных. Паттерн встроен в пример публичного API, которому будут следовать все пользователи.

```python
total = await Product.all().count()
items = await Product.all().order_by("-created_at").offset(pagination.offset).limit(pagination.limit).values()
```

**Рекомендация:**
Обновить docstring, чтобы показать комбинированный подход. Tortoise-ORM поддерживает параллельное выполнение обоих запросов: `total, items = await asyncio.gather(Product.all().count(), Product.all().order_by("-created_at").offset(pagination.offset).limit(pagination.limit))`. Задокументировать этот паттерн как рекомендуемый подход и отметить компромиссы.

### [HIGH] [Security] — AuthTomlConfig.cookie_secure по умолчанию False, допуская перехват сессии через HTTP

**Файл:** `config.py:29`
**Серьёзность:** Высокая

**Описание:**
`cookie_secure: bool = False` означает, что когда разработчик включает cookie-аутентификацию через `Core(app, auth="cookie")` и не устанавливает явно `cookie_secure=True` в `forgeapi.toml`, сессионная cookie будет передаваться через обычный HTTP. Флаг Secure на сессионных cookie является основной защитой, принудительно применяемой браузером, от перехвата сессии на уровне сети. Хотя CookieStrategy выдаёт `logger.warning` при `secure=False`, это предупреждение может быть подавлено в конфигурации логирования по умолчанию.

```python
cookie_secure: bool = False
```

**Рекомендация:**
Изменить дефолтное значение на `cookie_secure: bool = True`. Добавить валидатор, который бросает `ForgeAPIConfigError`, если `cookie_secure=False` и режим отладки в приложении выключен. Как минимум, повысить предупреждение CookieStrategy до блокирующей запуск ошибки в не-debug режиме.

### [MEDIUM] [Correctness] — Нет валидации того, что значения PaginationConfig являются положительными целыми числами

**Файл:** `config.py:33-35`
**Серьёзность:** Средняя

**Описание:**
`PaginationConfig` допускает любое `int` для `default_limit` и `max_limit`, включая ноль и отрицательные значения. Запись в `forgeapi.toml` вида `default_limit = -1` или `max_limit = 0` проходит валидацию Pydantic и сохраняется без ошибки. Когда `Core.__init__` вызывает `Paginator.configure()` с этими значениями, `Paginator.configure()` действительно бросает `ValueError`, но сообщение об ошибке будет ссылаться на внутренний код, а не на строку конфигурационного файла TOML пользователя.

```python
class PaginationConfig(BaseModel):
    default_limit: int = 20
    max_limit: int = 100
```

**Рекомендация:**
Добавить ограничения полей Pydantic: `default_limit: int = Field(20, ge=1)` и `max_limit: int = Field(100, ge=1)`. Также добавить `model_validator`, чтобы обеспечить выполнение условия `default_limit <= max_limit` с информативным сообщением об ошибке, ссылающимся на `forgeapi.toml`.

### [MEDIUM] [Correctness] — Нет верхней границы для номера страницы — риск переполнения целых чисел в арифметике смещения

**Файл:** `pagination/paginator.py:53-59`
**Серьёзность:** Средняя

**Описание:**
Хотя `page` ограничен `ge=1` через `Query()`, верхней границы нет. Запрос с `?page=2147483647&limit=100` производит `offset = (2147483647 - 1) * 100 = 214748364600`. Python обрабатывает это как целое число произвольной точности, но большинство SQL-баз данных либо выдадут ошибку переполнения целых чисел (PostgreSQL: `ERROR: value 214748364600 is out of range for type integer`), либо молча обернут значение, возвращая неожиданные результаты. Ошибка распространяется как необработанный 500 от драйвера базы данных.

```python
self.offset = (page - 1) * self.limit
```

**Рекомендация:**
Добавить ограничение максимальной страницы в параметре Query: `page: int = Query(1, ge=1, le=10000)`. Альтернативно, вычислять `max_page` из общего числа записей после запроса COUNT и возвращать 404 или пустой результат, когда страница превышает `max_page`. Задокументировать выбранный подход.

### [MEDIUM] [Design] — BaseUpdateSchema не предоставляет механизма для принудительного применения семантики частичного обновления

**Файл:** `schemas/base.py:33-41`
**Серьёзность:** Средняя

**Описание:**
`BaseUpdateSchema` наследует от `BaseModel` без конфигурации. Он задокументирован как база для PATCH-нагрузок, где «все поля должны быть Optional», но это ограничение применяется только документацией — нет `model_validator` в Pydantic, проверки `__init_subclass__` или инспекции метакласса, которые бы предупреждали или выдавали ошибку, если подкласс объявляет обязательные (не-Optional) поля. Разработчики, следующие неправильному паттерну, создадут PATCH-эндпоинт, требующий присутствия всех полей, молча нарушая семантику частичного обновления.

```python
class BaseUpdateSchema(BaseModel):
    """Base schema for PATCH (update) payloads.

    All fields should be ``Optional`` ...
    """
```

**Рекомендация:**
Добавить метод класса `__init_subclass__`, который инспектирует поля модели и бросает `TypeError` во время определения класса, если какое-либо поле имеет обязательное значение, отличное от `None`. Альтернативно, настроить `model_config = {"extra": "ignore"}` и более заметно задокументировать требование. Как минимум, добавить проверку во время выполнения в `model_validator` Pydantic.

### [MEDIUM] [Security] — BaseAppSettings может логировать или раскрывать чувствительные значения полей через repr/str

**Файл:** `settings/base.py:4-28`
**Серьёзность:** Средняя

**Описание:**
`BaseAppSettings` наследует от `pydantic_settings.BaseSettings` без настройки `secrets_dir`, `case_sensitive` или маскировки в `__repr__`. Pydantic BaseSettings по умолчанию включает все значения полей в `__repr__` и `__str__`. Если подкласс добавляет `jwt_secret: str` или `database_url: str`, логирование `settings` в отладочном выражении выведет сырой секрет или DSN (включая пароль) в лог-файлы. Это особенно актуально, поскольку пример в docstring forgeapi показывает именно эти чувствительные поля.

```python
class BaseAppSettings(BaseSettings):
    debug: bool = False
    app_name: str = "FastAPI App"

    model_config = SettingsConfigDict(
        env_file=".env",
        env_file_encoding="utf-8",
        extra="ignore",
    )
```

**Рекомендация:**
Добавить `@field_validator('*', mode='wrap')` или переопределить `__repr__` для маскировки полей, имена которых содержат 'secret', 'password', 'token' или 'key'. Альтернативно, настроить pydantic_settings с кастомным типом `SecretStr` для чувствительных полей и задокументировать, что пользователи должны объявлять секреты как `SecretStr`, а не `str` в своих подклассах.

### [MEDIUM] [Correctness] — load_config молча игнорирует неправильные типы секций TOML

**Файл:** `config.py:49-63`
**Серьёзность:** Средняя

**Описание:**
`load_config` передаёт `raw.get("project", {})` напрямую в конструктор Pydantic. Если TOML содержит `project = "not-a-table"` (скаляр вместо таблицы), `raw.get("project")` возвращает строку, и `KitConfig(project="not-a-table")` бросает `ValidationError` Pydantic. Эта ошибка не обрабатывается и будет отображаться как внутренний Python traceback, а не как понятная пользователю `ForgeAPIConfigError`. Более тонко: дополнительные ключи в секции (например, `[auth]` с неизвестным ключом) молча отбрасываются, потому что `AuthTomlConfig` использует строгий режим Pydantic, отключённый по умолчанию.

```python
return KitConfig(
    project=raw.get("project", {}),
    structure=raw.get("structure", {}),
    ...
)
```

**Рекомендация:**
Обернуть конструкцию `KitConfig(...)` в `try/except ValidationError` и пробрасывать как `ForgeAPIConfigError` с путём TOML и деталями валидации. Включить `extra="forbid"` на каждой модели конфигурации, чтобы неизвестные ключи в `forgeapi.toml` отображались как действенные ошибки, а не как молчаливая потеря данных.

### [MEDIUM] [Correctness] — _build_telegram передаёт пустую строку токена [''] когда BOT_TOKEN не установлен

**Файл:** `kit.py:192`
**Серьёзность:** Средняя

**Описание:**
Когда переменная окружения `BOT_TOKEN` отсутствует, `tokens` является пустым списком, и запасной вариант `kwargs["bot_token"] = tokens or [""]` передаёт список, содержащий одну пустую строку, в `TelegramStrategy`. `TelegramStrategy.__init__` фильтрует пустые токены через `tokens = [t for t in tokens if t]` и выдаёт предупреждение, но инициализатор Core не бросает ошибку. Приложение запускается без рабочей стратегии Telegram и молча отклоняет все Telegram-аутентификации с ошибкой 401.

```python
tokens = [t.strip() for t in raw.split(",") if t.strip()]
kwargs["bot_token"] = tokens or [""]
```

**Рекомендация:**
Бросать `ForgeAPIConfigError`, когда не найдено ни одного валидного токена: `if not tokens: raise ForgeAPIConfigError("BOT_TOKEN environment variable is not set.", hint="Set BOT_TOKEN=<your-bot-token> before starting the server.")`. Убрать запасной вариант `or [""]`.

### [MEDIUM] [Correctness] — Параметр 'logging' в kit.py затеняет модуль стандартной библиотеки logging

**Файл:** `kit.py:68`
**Серьёзность:** Средняя

**Описание:**
`Core.__init__` объявляет параметр с именем `logging: bool = True`. Внутри тела метода имя `logging` теперь ссылается на булев аргумент, а не на модуль стандартной библиотеки `logging`. Хотя модуль stdlib импортируется на уровне модуля, а логгер захватывается в переменную уровня модуля (`logger = logging.getLogger(...)`), любой код, добавленный внутри `__init__`, который пытается использовать `logging.getLogger()` или `logging.DEBUG`, завершится с `AttributeError: 'bool' object has no attribute 'getLogger'`. Это скрытая ошибка, ожидающая следующего добавления функционала.

```python
def __init__(
    self,
    app: FastAPI,
    *,
    ...
    logging: bool = True,
    ...
) -> None:
```

**Рекомендация:**
Переименовать параметр в `access_log: bool = True` или `request_logging: bool = True`, чтобы избежать затенения модуля stdlib. Обновить docstring и всех вызывающих соответственно.

```python
class ForgeAPIError(Exception):
```

**Рекомендация:**
Добавить опциональный параметр `status_code: int = 500` в `ForgeAPIError.__init__` и установить соответствующие значения по умолчанию для подклассов (`ForgeAPIConfigError` → 500, `ForgeAPIImportError` → 501). Зарегистрировать глобальный обработчик исключений FastAPI в Core, который преобразует `ForgeAPIError` в структурированный JSON-ответ. Убедиться, что поле `hint` никогда не раскрывается API-клиентам.

### [LOW] [Correctness] — _pluralize некорректно обрабатывает слова, оканчивающиеся на 's', и слова, оканчивающиеся на 'y', которым предшествует гласная

**Файл:** `controllers/base.py:6-11`
**Серьёзность:** Низкая

**Описание:**
`_pluralize` содержит две ошибки. Во-первых, слова, оканчивающиеся на 's', возвращаются без изменений, поэтому контроллер с именем 'StatusController' даёт префикс '/statuss'. Во-вторых, слова, оканчивающиеся на 'y', всегда преобразуются путём отбрасывания 'y' и добавления 'ies' независимо от предшествующей буквы: 'DayController' → '/daies' вместо '/days'. Стандартное англоязычное образование множественного числа требует проверки, является ли буква перед 'y' гласной (оставить 'ys') или согласной (использовать 'ies').

```python
def _pluralize(name: str) -> str:
    if name.endswith("y"):
        return name[:-1] + "ies"
    if not name.endswith("s"):
        return name + "s"
    return name
```

**Рекомендация:**
Расширить логику образования множественного числа: для слов, оканчивающихся на 'y', проверять, входит ли `name[-2]` в `'aeiou'` — если да, добавлять 's' (days); иначе заменять 'y' на 'ies' (categories). Для слов, уже оканчивающихся на 's', добавлять 'es' (statuses) или разрешить явное переопределение префикса. Рассмотреть использование библиотеки `inflect` для корректности.

### [LOW] [Design] — BaseSchema раскрывает id как простой int без поддержки UUID или составного ключа

**Файл:** `schemas/base.py:15`
**Серьёзность:** Низкая

**Описание:**
`BaseSchema` объявляет `id: int` как обязательное поле. Это жёстко закодирует целочисленные первичные ключи как единственный поддерживаемый тип идентификации. Приложения, использующие UUID первичные ключи (распространено для распределённых систем во избежание перебора последовательных ID) или составные ключи, не могут наследоваться от `BaseSchema` без переопределения поля `id` и потенциальной путаницы в валидаторе Pydantic. Режим `from_attributes` означает, что несоответствие обнаруживается только во время сериализации, вызывая непонятный `ValidationError`.

```python
id: int
created_at: datetime
updated_at: datetime
```

**Рекомендация:**
Изменить `id` на `id: int | str`, чтобы поддерживать как целочисленные, так и UUID первичные ключи, или использовать `from typing import Any; id: Any`. Альтернативно, убрать `id` из `BaseSchema` и позволить подклассам объявлять свой собственный тип первичного ключа. Чётко задокументировать ожидаемую структуру модели.
